## Question 5 of 9: How do you prioritize system upgrades and optimize ongoing system design? Can you share your approach to strategic planning and risk analysis?

A security vulnerability obviously gets handled immediately. A framework upgrade that would be nice but doesn't unlock anything that can usually wait. Spend more time on prep work than the actual work, and that's on purpose. If you plan the upgrade well, the execution is the easy part. At Trenum, Revit released new versions every year, and my plugins had to keep working across them. So I built automated tests specifically so I could upgrade with confidence instead of guessing what might break. I'm a big believer in making changes reversible when you can. So feature branches, automated tests, containerized environments. At Scripthammer, I've containerized development with Docker, and I use Playwright for end-to-end browser testing. So I can test upgrades in isolation and verify nothing broke before it touches anything real. If an upgrade goes wrong, it should be easy to roll back. If it's not easy to roll back, then that tells you the change needs more planning before you start. I try to avoid big rewrites unless there's no other option. I prefer incremental improvements, swap out one piece, verify it works, and then move on to the next. So smaller changes, smaller risk, faster feedback.
